4. Audio (Most Important — Study This)

This is the core of Munsit. Expect a coding task or deep questions here. You built AudioDemoScreen.tsx — know every line.
Hard Walk me through implementing audio recording in Expo/React Native.

Using expo-audio (SDK 53+ replacement for the deprecated expo-av) — five steps:

  1. Request permission — requestRecordingPermissionsAsync()
  2. Configure AVAudioSession (iOS critical) — setAudioModeAsync({ allowsRecording: true, playsInSilentMode: true }). Without this, recording fails silently on iOS.
  3. Prepare + start — await recorder.prepareToRecordAsync() then recorder.record(). Must call prepare before every record.
  4. Real-time metering — useAudioRecorderState(recorder, 100) gives state.metering (dBFS) every 100ms for visualisation
  5. Stop + play — await recorder.stop() → recorder.uri has the file. Load it with player.replace({ uri }) then player.play()

Reset audio mode after stop (allowsRecording: false) so playback routes to the speaker, not the earpiece.

Full Recording + Playback Code (expo-audio — SDK 56)

import {
  requestRecordingPermissionsAsync,
  setAudioModeAsync,
  useAudioRecorder,
  useAudioRecorderState,
  useAudioPlayer,
  useAudioPlayerStatus,
  RecordingPresets,
} from 'expo-audio';

// ── Hooks (top level of component) ──────────────────────────────────────────
const recorder = useAudioRecorder({ ...RecordingPresets.HIGH_QUALITY, isMeteringEnabled: true });
const recorderState = useAudioRecorderState(recorder, 100); // polls every 100ms
const player = useAudioPlayer(null);           // start with no source
const playerStatus = useAudioPlayerStatus(player);

// ── 1. Permissions ───────────────────────────────────────────────────────────
const { granted } = await requestRecordingPermissionsAsync();
if (!granted) return;

// ── 2. Configure iOS audio session ───────────────────────────────────────────
await setAudioModeAsync({ allowsRecording: true, playsInSilentMode: true });

// ── 3. Prepare + start ───────────────────────────────────────────────────────
await recorder.prepareToRecordAsync();   // MUST call before every record()
recorder.record();

// ── 4. Read amplitude from state (in a useEffect watching recorderState) ─────
const amplitude = recorderState.metering !== undefined
  ? Math.max(0, (recorderState.metering + 60) / 60)  // dBFS -60→0 normalised to 0→1
  : 0;

// ── 5. Stop + reset audio mode ────────────────────────────────────────────────
await recorder.stop();
await setAudioModeAsync({ allowsRecording: false, playsInSilentMode: true });
// recorder.uri now has the local file path

// ── 6. Load into player and play ─────────────────────────────────────────────
player.replace({ uri: recorder.uri! });  // replace() loads new source
player.play();

// ── 7. Watch playback status ─────────────────────────────────────────────────
// playerStatus.currentTime, playerStatus.duration, playerStatus.playing
// playerStatus.isLoaded — wait for this before reading duration

// ── 8. Cleanup ────────────────────────────────────────────────────────────────
// useAudioRecorder and useAudioPlayer clean up automatically on unmount
// No manual unloadAsync() needed — lifecycle is managed by the hooks
Hard What is AVAudioSession and why does it matter?

AVAudioSession is iOS's audio routing and categorisation system. It tells the OS what your app is doing with audio so it can make decisions about interruptions, routing (speaker vs earpiece vs Bluetooth), and background behaviour.

In expo-audio you configure it with setAudioModeAsync(). If you don't set allowsRecording: true before recording, iOS won't route mic input to your app even if permission is granted. If you don't set playsInSilentMode: true, audio fails on a device in silent mode. After recording you must reset to allowsRecording: false otherwise playback routes through the earpiece instead of the speaker.

On Android the equivalent is AudioManager focus — expo-audio wraps both platforms with the same API.

expo-av vs expo-audio: expo-av was deprecated in SDK 53. expo-audio has a hook-based API (useAudioRecorder, useAudioPlayer) and handles lifecycle automatically — no manual unloadAsync() on unmount. expo-av's EXEventEmitter.h header was removed from ExpoModulesCore in SDK 56, so expo-av won't compile in SDK 56.

Hard How would you stream audio chunks for real-time transcription?

Two approaches:

  1. WebSocket (bidirectional): Record in chunks (segment-based or time-based), send each chunk as binary over WebSocket. Server transcribes and pushes text back on the same connection. Best for low-latency.
  2. SSE + file upload: Upload the whole file, subscribe to SSE stream for progressive results. Simpler but not truly real-time.

Munsit likely uses WebSocket or Socket.IO for this. The recording side would use a native module (not expo-av directly) for chunk-by-chunk PCM access, or send the m4a file and let the server handle chunking.

5. Real-time: Socket.IO / SSE

Hard WebSocket vs SSE — when do you choose each?

SSE (Server-Sent Events): Server → client only. One HTTP connection. Auto-reconnects. Use when you only need to receive data (transcription results, live subtitles, feed updates).

WebSocket: Bidirectional, full-duplex. Use when the client also sends data over the same channel (sending audio chunks, heartbeats, control messages).

For Munsit specifically: Probably WebSocket — client sends audio, server pushes transcript words. Socket.IO wraps WebSocket and adds rooms, namespaces, reconnection, and fallback to polling.

SSE Consumer (React Native)

// React Native doesn't have EventSource — use fetch with streaming
async function subscribeToTranscript(sessionId: string, onWord: (word: string) => void) {
  const response = await fetch(`https://api.munsit.ai/sessions/${sessionId}/stream`, {
    headers: { Authorization: `Bearer ${token}` },
  });

  const reader = response.body!.getReader();
  const decoder = new TextDecoder();

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    const chunk = decoder.decode(value);
    // SSE format: "data: \n\n"
    const lines = chunk.split('\n').filter((l) => l.startsWith('data: '));
    for (const line of lines) {
      const payload = JSON.parse(line.slice(6));
      if (payload.word) onWord(payload.word);
    }
  }
}

6. Zustand

Easy How does Zustand differ from Redux Toolkit?

Redux Toolkit: Boilerplate-reduced Redux. Still actions, reducers, dispatch. Good for complex state with many reducers, middleware (RTK Query, logging), and dev tools.

Zustand: Direct mutation (via immer internally), no boilerplate, no Provider needed. Just a hook. Smaller bundle, simpler code. Better for localised feature state.

For Munsit they likely use Zustand for UI state (recording state, active transcription session) and React Query for server state (API calls, caching).

Zustand Store Pattern (what you built)

import { create } from 'zustand';

type Language = 'en' | 'ar';

interface TranscriptionStore {
  segments: Segment[];
  liveText: string;
  isRecording: boolean;
  language: Language;
  // actions
  appendWord: (word: string) => void;
  flushLive: () => void;
  setRecording: (v: boolean) => void;
  setLanguage: (lang: Language) => void;
  clear: () => void;
}

export const useTranscriptionStore = create((set, get) => ({
  segments: [],
  liveText: '',
  isRecording: false,
  language: 'en',

  appendWord: (word) =>
    set((s) => ({ liveText: s.liveText ? `${s.liveText} ${word}` : word })),

  flushLive: () => {
    const { liveText, language } = get(); // get() reads current state synchronously
    if (!liveText.trim()) return;
    set((s) => ({
      segments: [...s.segments, { id: Date.now().toString(), text: liveText, language }],
      liveText: '',
    }));
  },

  setRecording: (isRecording) => set({ isRecording }),
  setLanguage: (language) => set({ language }),
  clear: () => set({ segments: [], liveText: '', isRecording: false }),
}));

// Usage in component:
// const { liveText, appendWord } = useTranscriptionStore();
// Or select only what you need (avoids unnecessary re-renders):
// const liveText = useTranscriptionStore((s) => s.liveText);

7. React Query (TanStack Query)

Honest position: You know the model well, production experience is with RTK Query which is conceptually identical. Frame it that way.
Med Explain staleTime vs gcTime (cacheTime in v4)

staleTime: How long cached data is considered fresh. During this window, no background refetch happens. Default: 0 (always stale = always refetch on mount).

gcTime (formerly cacheTime): How long unused/inactive query data stays in memory before being garbage collected. Default: 5 minutes.

Example: staleTime: 60_000 means data fetched in the last minute is reused without a network call.

Med Common RTL pitfalls?
  • Icons that shouldn't flip — back arrows flip correctly, but checkmarks, play buttons shouldn't. Use transform: [{{ scaleX: isRTL ? -1 : 1 }}] to selectively prevent flip.
  • Hardcoded left/right margins — use marginStart/marginEnd instead of marginLeft/marginRight.
  • Absolute positioning — left: 0 doesn't flip automatically. Use start: 0 (React Native 0.65+).
  • Arabic fonts — system fonts on iOS/Android handle Arabic fine. Avoid custom fonts unless they support Arabic glyphs.
  • Numbers in Arabic text — Arabic-Indic vs Western numerals. Usually keep Western (0-9) for timestamps and counts.

9. Performance

Hard FlashList vs FlatList — explain the difference and when to use FlashList

FlatList mounts new React components as items scroll into view and unmounts them when they leave. The JS thread does this work. For short lists (<100 items) this is fine.

FlashList (from Shopify) recycles cell components — like RecyclerView on Android. It reuses existing component instances and updates their data. This means no mount/unmount overhead. 10x faster initial render on long lists.

Migration: Drop-in replacement. Change FlatList → FlashList, add estimatedItemSize prop (average item height in px). FlashList will warn you if the estimate is very wrong.

import { FlashList } from '@shopify/flash-list';

<FlashList
  data={transcriptions}
  renderItem={({ item }) => <TranscriptionRow item={item} />}
  keyExtractor={(item) => item.id}
  estimatedItemSize={64}       // required — average row height
  onEndReached={loadMore}
  onEndReachedThreshold={0.5}
/>
Hard Why is Reanimated faster than the built-in Animated API?

The built-in Animated API runs animation logic on the JS thread and communicates each frame to the UI thread over the bridge. If the JS thread is busy (rendering, API calls), animations stutter.

Animated with useNativeDriver: true moves the animation to the UI thread, but only works for transform and opacity.

Reanimated 2/3 uses Worklets — small JS functions compiled by Hermes to run directly on the UI thread. This means all animation logic runs on the UI thread with no bridge crossing. Smooth 60fps even when JS is busy. Supports any style property (not just transform/opacity).

Med How do you debug jank (dropped frames)?
  1. Flipper — React Native performance monitor shows JS and UI frame rates separately. Identifies which thread is dropping frames.
  2. React DevTools Profiler — shows which components re-render and how long they take.
  3. Instruments (iOS) — Time Profiler for native CPU, Allocations for memory leaks.
  4. Systrace / Perfetto (Android) — native trace with method timing.
  5. FPS Monitor in Dev Menu — quick check while using the app.

Common culprits: expensive renderItem, missing keyExtractor, large lists without FlashList, inline functions/objects creating new references on every render, heavy images not cached.

10. Expo EAS & OTA Updates

Med What is EAS Build and how does it differ from classic expo build?

expo build (deprecated) used Expo's managed build service but had limited customisation and forced you into Managed workflow.

EAS Build runs on Expo's cloud infrastructure but supports any project including bare React Native. You control the build environment via eas.json. Supports custom native code, local credentials, and can run your own scripts.

// eas.json
{
  "build": {
    "development": { "developmentClient": true, "distribution": "internal" },
    "preview": { "distribution": "internal" },
    "production": { "autoIncrement": true }
  }
}
Hard What is runtime version and why does it matter for OTA?

OTA (expo-updates) pushes a new JS bundle to users without a store release. But if your update requires new native code (a new Expo module, a new permission), and the user hasn't updated the app binary, the new JS will crash.

Runtime version ties a JS bundle to a specific native binary. When you add native code, you bump the runtime version. The OTA service only delivers updates to devices with a matching runtime version. Old binary = old JS. New binary = new JS.

// app.json
{
  "expo": {
    "runtimeVersion": {
      "policy": "appVersion"   // auto: runtime = app version
    }
  }
}

JS Core — var / let / const, Closures, Hoisting, Event Loop

Classic var vs let vs const — what are the real differences?
// VAR — function-scoped, hoisted, can be redeclared
var x = 1;
var x = 2;          // ✅ no error
console.log(y);     // undefined (hoisted, not ReferenceError)
var y = 5;
function test() {
  if (true) { var z = 10; }
  console.log(z);   // 10 — var leaks out of the block!
}

// LET — block-scoped, TDZ (Temporal Dead Zone), can't redeclare
let a = 1;
let a = 2;          // ❌ SyntaxError
console.log(b);     // ❌ ReferenceError — TDZ, not undefined
let b = 5;

    Array.isArray(item) ? [...acc, ...flatten(item)] : [...acc, item],
  []);

// Factorial
function factorial(n: number): number {
  if (n <= 1) return 1;           // base case — ALWAYS have one
  return n * factorial(n - 1);   // recursive case
}

// Tree traversal — common in navigation/menu structures
interface TreeNode {
  id: string;
  label: string;
  children?: TreeNode[];
}

function findNode(tree: TreeNode[], id: string): TreeNode | null {
  for (const node of tree) {
    if (node.id === id) return node;
    if (node.children) {
      const found = findNode(node.children, id);
      if (found) return found;
    }
  }
  return null;
}

// Fibonacci — shows memoisation optimisation
const memo = new Map();
function fib(n: number): number {
  if (n <= 1) return n;
  if (memo.has(n)) return memo.get(n)!;
  const result = fib(n - 1) + fib(n - 2);
  memo.set(n, result);
  return result;
}

// cacheMiddleware.ts — intercepts specific Redux actions and persists to MMKV
// This is the Interceptor pattern (like axios interceptors, but for Redux)

const inclusiveCache = [...userReducerCacheActions, ...appReducerCacheActions];

export const cacheMiddleware: Middleware = () => next => (action: any) => {
  // Intercept the action BEFORE it reaches the reducer
  if (inclusiveCache.includes(action.type)) {
    if (action.payload !== null) {
      storage.set(action.type, JSON.stringify(action.payload)); // persist to MMKV
    } else {
      storage.delete(action.type); // clear cache on null
    }
  }
  next(action); // pass through to reducer
};

How to Say It In the Interview

How do you architect a React Native app?

"I use feature-based modular architecture — folders organised by domain, not by file type. Inside each feature I follow MVVM: the screen component is the View and handles only UI, the custom hook is the ViewModel with all business logic and state, and the service layer plus Redux slices are the Model. Nothing touches axios directly from a screen or a hook — API calls live in service files that are independently testable and reusable."

"I also use a custom Redux middleware for caching critical state to MMKV so it persists across restarts — that's the Interceptor pattern, same concept as axios interceptors but for Redux actions."

"In myosntv that structure handles 15+ Redux slices, 4 separate axios instances for different backends including Kontent.ai, PayFacade, and the main API — and it scales without becoming a mess because each feature owns everything it needs."
